off-by-one漏洞

简介

off-by-one 漏洞是一种特殊的溢出漏洞，off-by-one 指程序向缓冲区中写入时，写入的字节数超过了这个缓冲区本身所申请的字节数并且只越界了一个字节。这篇文章仅讨论在堆空间下的off-by-one漏洞。

产生原因：

off-by-one 是指单字节缓冲区溢出，这种漏洞的产生往往与边界验证不严和字符串操作有关，通常包括：

  • 使用循环语句向缓冲区中写入数据时，循环的次数设置错误导致多写入一个字节
  • 字符串操作不合适，比如忽略了字符串末尾的\x00

利用思路：

溢出字节为可控制任意字节：通过修改大小造成块结构之间出现重叠，从而泄露其他块数据，或是覆盖其他块数据。也可使用 NULL 字节溢出的方法

溢出字节为 NULL 字节：在 size 为 0x100 的时候，溢出 NULL 字节可以使得 prev_in_use 位被清，这样前块会被认为是 free 块。（1） 这时可以选择使用 unlink 方法（见 unlink 部分）进行处理。（2） 另外，这时 prev_size 域就会启用，就可以伪造 prev_size ，从而造成块之间发生重叠。此方法的关键在于 unlink 的时候没有检查按照 prev_size 找到的块的大小与prev_size 是否一致。

最新版本代码中，已加入针对 2 中后一种方法的 check ，但是在 2.28 及之前版本并没有该 check

保护情况：

程序功能：

一个简单的heap菜单题,可以创建任意大小的堆，其指针保存在数组上，并将数据输入到堆中储存，可以展示堆内的内容、修改堆内的数据、删除创建好的堆，最多可以创建10个堆。

heap结构：

heaparry一共有十位，里面保存着指向heap的heap指针；heap占0x10个字节，其中第一位是content的大小，第二位是指向content的指针；content里面储存着用户输入的内容，大小由用户设定。如下图所示：

漏洞点：

在创建堆的函数中，向堆写入数据时多写入了一个数据，典型的off-by-one漏洞（不过本题漏洞点构造的较为生硬——直接让size + 1）。

漏洞利用思路：

总体思路：

利用off-by-one漏洞改写chunk的size大小，使chunk重叠，然后释放掉这个chunk，让新创建的chunk申请到这个原chunk的heap位置，将heap指针的地址改写成atoi函数的got表地址，然后查看该chunk的内容，就可以得到atoi函数的地址，然后利用编辑功能将该位置改写成system地址，最后输入‘sh’即可getshell。

具体分解：

先创建一个content大小为0x18的heap和一个content大小为0x10的heap，分别称为heap0，heap1，这样设置的目的是为了使heap0的content0使用heap1_chunk的pre_size位0x8大小的空间，这样我们可以溢出到heap1_chunk的size域：

可以看到两个content_chunk被分配到的实际大小都是0x20。接下来利用off-by-one漏洞将heap1_chunk的size域覆盖为0x41：

可以发现heap1的heap1_chunk不见了，只剩下0x40大小的content1_chunk了。然后我们利用菜单功能删除heap1：

发现heap1_chunk和content1_chunk都被放入了fastbin中。这时候我们再创建一个content大小为0x30的heap（这样heap_chunk就会申请到原来heap1_chunk的空间，content_chunk就会申请到大小为0x40的content1_chunk的空间，这样就造成了content_chunk位于heap_chunk的上方，使用户输入的数据可以覆盖heap_chunk），需要注意的是由于删除heap时会将heaparry对应的位置置空，而创建heap时是循环查看heaparry每个位是否为空，如果不为空就在这个位置创建heap，所以最新创建的这个heap仍应该称为heap1（在heaparry[1]的位置）：
